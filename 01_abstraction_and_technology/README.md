# 📖 [컴퓨터 구조] 추상화와 기술

---

## 1. 고장 감내와 시스템 생존 기술
> 대형 임베디드 기기(자동차, 통신 장비 등)에서 치명적인 에러로부터 시스템을 보호하는 하드웨어와 소프트웨어의 협업 아키텍처

#### 🚨 하드웨어 중복과 그 한계 극복
* **듀얼 코어 락스텝**: 2개의 코어가 동일 연산을 수행하고 비교기로 검증. 단, 공통 원인 고장(`CCF`)을 막기 위해 두 코어 간에 시간적/공간적 분리를 두어 엇박자로 실행하도록 설계한다.
* **Watchdog Timer(`WDT`)** : 커널 패닉이나 무한 루프로 소프트웨어가 멈췄을 때, 드라이버가 타이머를 초기화하지 못하면 하드웨어 리셋 핀을 때려 시스템을 강제 재부팅하는 시간적 여유분 기술.

#### 💡 ECC 메모리: 신드롬과 오류 정정
> 데이터 저장 시 여유 비트(`Parity`)를 추가하여, 읽을 때 발생하는 에러를 감지하고 복구한다.

* **1비트 에러 (`CE` - Correctable Error)**: 하드웨어가 신드롬 주소를 바탕으로 나노초 단위에서 스스로 비트를 뒤집어(`XOR`) 자동 수정. OS에는 인터럽트를 주지 않고 카운트만 올리며, 커널은 이를 폴링하여 장비 노후화 로그만 남긴다.
* **2비트 에러 (`UE` - Uncorrectable Error)**: 하드웨어 복구 불가능. 즉각적으로 `NMI`를 발생시키며, 드라이버는 커널 패닉을 유발하거나 해당 프로세스에 `SIGBUS`를 날려 시스템 오작동을 강제로 차단한다.

## 2. NMI 핸들러와 커널 드라이버
> 최상위 하드웨어 인터럽트인 `NMI`는 일반적인 `request_irq()`로 제어할 수 없으며, 커널 코어를 수정하지 않고 드라이버 단에서 합법적으로 훅을 걸어 제어한다.

```c
#include <asm/nmi.h>

// 1. 치명적 하드웨어 에러 발생 시 동작할 콜백 로직 구현
static int my_hw_err_handler(unsigned int val, struct pt_regs *regs) {
    pr_emerg("하드웨어 ECC 2비트 에러 발생! 시스템 안전 모드 전환.\n");
    return NMI_HANDLED;
}

// 2. 콜백 맵핑 후 드라이버 로드 시(Init) 커널 NMI 프레임워크에 리스트로 등록
static struct nmi_desc my_nmi_desc = { .handler = my_hw_err_handler, .name = "my_err_handler" };
register_nmi_handler(NMI_LOCAL, &my_nmi_desc);
```

#### 🔍 아키텍처별 핵심 커널 소스 경로
* x86: `arch/x86/kernel/nmi.c`
* ARM64: `drivers/irqchip/irq-gic-v3.c`

## 3. 하드웨어 성능 지표와 전력 장벽(`DVFS`)
> 소프트웨어 엔지니어가 하드웨어 성능을 수치로 분석해야 하는 이유

#### ⚡ 실행 시간 공식
`실행 시간 = 명령어 수(IC) × CPI × 사이클 타임(Cycle Time)`
* 비율 계산의 함정: 명령어 수가 `0.6`배 줄고 CPI가 `1.1`배 늘었다면, 실행 시간은 기존의 `0.6 × 1.1 = 0.66배`가 된다. 하지만 성능 향상도는 역수인 `1 / 0.66 = 약 1.51배` 빠르다고 표현해야 한다. (컴파일러 최적화 옵션 `-Os` vs `-O2`의 트레이드오프)

#### 🔋 전력 공식과 DVFS
`전력(Power) ∝ 1/2 × C × V² × f`
* 전력은 전압($V$)의 제곱에 비례한다. 배터리와 발열을 잡기 위해 리눅스 `cpufreq` 드라이버는 단순히 클럭 주파수($f$)만 낮추는 것이 아니라, 낮아진 주파수에서 버틸 수 있는 최소 전압($V$)까지 동적으로 끌어내려 기하급수적인 전력 절감을 달성한다.

## 4. 암달의 법칙
> 전체 성능의 향상은 개선된 부분이 전체에서 차지하는 비중(비율)에 의해 절대적으로 제한받는다.

* 현업 활용: 80%의 시간을 차지하는 무거운 커널 로직을 놔두고, 20%짜리 UI 로직을 10배 최적화해봤자 전체 성능 향상은 미미하다. 멀티코어 도입 시 병렬화 불가능한 순차 코드의 비율 때문에 늘릴 수 있는 성능 한계를 계산하고 방어하는 보고서의 핵심 논리

## 5. 하드웨어 성능 카운터(PMU)와 병목 추적
> 코드에 소프트웨어 타임을 심는 것은 오버헤드가 발생한다. CPU 내부의 `PMU` 레지스터를 활용하면 0% 오버헤드로 정확한 통계를 추출할 수 있다.

#### 🛠️ 실무 분석 툴: `perf`
> 리눅스 시스템 엔지니어의 핵심 병목 추적 도구. `CPI(IPC)`를 파악하고, `Cache-misses`(메모리 접근 병목)나 `Branch-misses`(분기 예측 실패 병목)를 데이터 기반으로 색출한다.

```bash
perf stat ./my_heavy_program
```

##### 🎯 타겟 보드(임베디드)에 perf가 없을 때의 생존 전술
> 경량화된 타겟 리눅스 환경에서 프로파일링 툴이 누락되었을 때의 대응 파이프라인

1. 소프트웨어 쪼개기: `clock_gettime(CLOCK_MONOTONIC)` 함수를 의심 구간 앞뒤로 덮어씌워 소요 시간(ns 단위)을 직접 도출한다.
2. 크로스 컴파일 밀어 넣기: `Yocto/Buildroot` 환경에서 `CONFIG_PERF_EVENTS` 커널 옵션을 켜고, 타겟용 `perf` 바이너리를 빌드하여 보드 `/tmp`에 직접 적재.
3. PMU 직접 제어: ARM 매뉴얼을 참조, `mrs %0, pmccntr_el0` 어셈블리를 삽입한 초소형 커널 모듈(드라이버)을 작성해 직접 하드웨어 카운터를 로깅한다.
